# ML-accelerated Supernova Modeling

Supernovae are explosions that massive stars undergo at the end of their life. Over millions of years, many of these explosions 
continually blow away the gas that make up galaxies, preventing them from collapsing in on themselves under the pull of gravity.
However, the detailed geometries and sharp gradients of these explosions make simulating them a computationally demanding task. 
Moreover, as galaxies are made of millions to billions of stars, it becomes prohibitive to trace many individual supernovae 
in a simulation of an entire galaxy. 

Therefore, the need for fast and inexpensive models of supernovae are critical for simulations of galaxies. Here, we use deep
learning to accelerate supernova modeling, by training a Fourier Neural Operator (FNO) on many examples of pre-simulated supernovae.
This codebase enables training, evaluation, and analysis of predictions of our deep learning model.

## Quickstart

### 1. Environment setup

```bash
# Option A: Conda
conda create -n supernova python=3.11
conda activate supernova
pip install torch==2.8.0 torchvision torchaudio --index-url https://download.pytorch.org/whl/cu128
pip install -r requirements.txt

# Option B: Docker
docker build -t supernova .
```

Note that running the training scripts requires access to an NVIDIA GPU. Note also that if installing using conda, PyTorch is installed separately from those listed in requirements.txt.

### 2. Data download

This codebase uses the [supernova_explosion_64](https://github.com/PolymathicAI/the_well) dataset from The Well benchmark suite ([Ohana et al. 2024](https://arxiv.org/abs/2412.00568)). Data was generated by Hirashima et al. 2023[a](https://arxiv.org/abs/2302.00026), [b](https://arxiv.org/abs/2311.08460).

To download a small subset of the data, run the following:
```bash
the-well-download --base-path basedata --dataset supernova_explosion_64 --split train --first-only
the-well-download --base-path basedata --dataset supernova_explosion_64 --split valid --first-only
```

The `--first-only` flag downloads only the first file of each split, ideal for a demo run. Combined this amounts to about 7 GB. The
full dataset is much larger (~300 GB).

### 3. Configure an Experiment

From your working directory, edit the paths.base_data field to point from the project directory to the location in which you stored your
data. (If using Docker, there is no need to touch this field).

Edit `configs/params.yaml` to add a named experiment under the `fno:` block, or create a file in `configs/indiv/<experiment_name>.yaml`. 
Only parameters that differ from the `fno.default` block need to be specified:

```yaml
# configs/indiv/my_experiment.yaml
model:
  n_modes: 16
  n_layers: 4
training:
  batch_size: 2
  max_epochs: 4
  epoch_save: 1
validation:
  freq: 2
  batch_size: 3
preprocess:
  norm_samples: 100
  norm_cycle: 1
base_split:
  seed: 14383421
  proportion: 0.4
```

If running on a local machine, we recommend using the sample parameters above for a non-computationally intensive run. The above parameters 
are known to work on an NVIDIA RTX 3070 with 8 GB VRAM. A sample YAML file has already been placed in the configs/indiv folder.

### 4. Run the model

If using Docker, first run:

Windows:
```cmd
docker run --gpus all --shm-size=8g --rm -it -v %cd%:/app supernova
```

Linux:
```bash
docker run --gpus all --shm-size=8g --rm -it -v $(pwd):/app supernova
```

to open an interactive shell within the container.

**1. Preprocess the data.**

Extract global statistics used for data preproessing (e.g., statistics for normalizing the data).

**2. Train the model.**

```bash
python run.py -r my_experiment
```

The benchmarks can be found within `models/my_experiment` (or whatever paths are configured in `configs/configs.yaml`).

**3. Evaluate the model.**

Produce evaluation statistics of your run and automatically select the best model.

```bash
python evaluate.py -r my_experiment
```

Performance summary can be found in `plots/my_experiment/fno_evals.png` (or configured path).

**4. Produce analysis plots.**

```bash
python analysis_explore.py -r my_experiment
```

Plots can be found in `plots/my_experiment` (or configured path). Note that a demo run is unlikely to yield good results, as training a good model requires much more compute time.

## Advanced Run Options

### Options

`run.py` may be run with additional flags, listed below.

- **`-r` / `--experiment-name`**: Name of the experiment run. Controls which parameters are loaded and where outputs are saved. Default `default`.
- **`-m` / `--mode`**: Run mode. One of `dev`, `live`, `series`, `sweep`. Default `dev`. `dev` runs interactively with no WandB logging. `live` runs interactively with WandB. `series` is for non-interactive SLURM submissions with WandB. `sweep` is for WandB hyperparameter sweeps.
- **`-e` / `--default-epoch`**: Epoch number to load when restarting from a checkpoint. If omitted during a restart, defaults to the most recent benchmark epoch.
- **`-R` / `--restart`**: If set, resumes training from an existing checkpoint rather than starting fresh.
- **`-d` / `--data-path`**: Override the default dataset path specified in `config.yaml`.

### Adjusting Config and Params

**`configs/config.yaml`**

Static project settings: dataset metadata, directory paths, W&B project/entity names. Edit paths to match your environment.

**`configs/params.yaml`**

Hyperparameters organized under `fno`, with a `default` block and named experiment blocks. Parameters merge as: **default → experiment → restart file** (highest priority).

Key parameter groups:

| Group | Key parameters | Notes |
|-------|---------------|-------|
| `model` | `n_modes`, `hidden_channels`, `n_layers`, `factorization`, `factorize_rank` | FNO architecture |
| `training` | `learn_rate`, `weight_decay`, `batch_size`, `max_epochs`, `epoch_save`, `grad_max`, `lr_decay_period`, `lr_decay_gamma` | Optimizer and schedule |
| `validation` | `freq`, `batch_size` | Validation subsampling and batch size |
| `preprocess` | `log`, `asinh`, `norm_samples`, `norm_cycle` | For data preprocessing |
| `base_split` | `seed`, `proportion` | Train/holdout split |
| `physics` | `tune_start`, `lambda_idg`, `lambda_consv_mass`, `lambda_consv_momentum` | Physics loss weights and schedule |
| `autoregression` | `order` | Multi-step training horizon |
| `logging` | `steps_per_loss`, `steps_per_grad`, `epochs_per_eval` | W&B logging frequency |

**NOTICE**: **<u>Refrain from setting `training.batch_size`, `validation.batch_size` and `autoregression.order` to either 6 or 64.</u>** This codebase uses 
automatic axis detection based on the tensor shape to identify the field and box size axes, which are set in `configs/config.yaml: data_summary`. 
For this dataset, they are 6 and 64 respectively. To prevent silent downstream errors caused by a shape collision, if any of these parameters are set 
to these values, the parameter reader will raise an error before the training begins.

### WandB Setup

If logging using WandB, update your `wandb_project` and `wandb_entity` in their respective
fields in `configs/config.yaml`.

If performing a hyperparameter sweep with WandB, set the mode `-m` / `--mode` to `sweep`,
and configure the sweep yaml to run `run.py` with the appropriate flags.

Note that in `sweep` mode the training script draws its run name
and parameters from the sweep config on WandB, so that the user does not set any
parameters.

### Slurm

`submit.py` allows for job submission on the HPCC at University of California, Riverside. 

To configure, place the following in `configs/configs.yaml`:
- the conda environment for your job in `compute.conda_env`
- your email in `names.email`

If submitting a job using this script, use the following flags:

- **`-m` / `--mode`**: Run mode. One of `live`, `series`, `sweep`. Default `live`. Passed to `run.py`; see the Options
section for details on each mode.

- **`-r` / `--run-names`**: Name of the experiment run(s). `live` and `series` mode only. Controls which parameters are loaded and where outputs are saved. Default `default`. If using `series` mode, multiple run names may be specified, e.g., `-r my_run_1 my_run_2 my_run_3`.

- **`-s` / `--sweep-id`**: W&B sweep ID. Required for `sweep` mode.
- **`-n` / `--nruns`**: Number of sweep agents to submit. Only valid for `sweep` mode. Default `1`.

- **`-e` / `--default-epoch`**: Epoch number to load when restarting from a checkpoint. If omitted during a restart, defaults to the most recent benchmark epoch. Passed to `run.py`. Use with `live` mode only.
- **`-R` / `--restart`**: If set, resumes training from an existing checkpoint rather than starting fresh. Passed to `run.py`. Use with `live` mode only.

- **`-c` / `--copy-data-to-scratch`**: Copies data to the HPCC scratch disk before running. As of April 2026, omitting
this flag causes a significant training slowdown as training data must be read over NFS. Copying the whole dataset to
the scratch disk first adds a roughly 25-minute overhead per job on HPCC, which is negligible for multi-day training runs.

## Data

**Dataset:** `supernova_explosion_64` from [The Well](https://github.com/PolymathicAI/the_well) benchmark suite.

| Property | Value |
|----------|-------|
| Fields | Density, pressure, temperature, vx, vy, vz |
| Spatial resolution | 64 × 64 × 64 |
| Timesteps | 59 |
| Dimensions | 3D |
